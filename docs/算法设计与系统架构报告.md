# `path-planner` 算法设计与系统架构报告

生成日期：2026-06-06

## 1. 资料边界

本报告面向根仓库当前使用的子项目 `path-planner`，而不是旧的
`a_gcs_ws-2.0.1-direction-cone-curvature` 实验目录。报告依据如下：

- `path-planner/src/path_planner/`
- `path-planner/tests/`
- `path-planner/docs/superpowers/specs/`
- 根仓库与 `path-planner` 之间的 JSON 契约

当前 `path-planner` 的执行层边界是 2D 几何路径规划与诊断。它可以评估 sampled
geometric candidate 的碰撞、地形代价、方向锥约束和曲率/航向风险，但不宣称输出
Ackermann-feasible trajectory。

## 2. 总体结论

`path-planner` 当前是一个分层路径规划与执行层评估原型。稳定主链路仍然是：

```text
path-planner-request/v1
  -> cost/passable grid
  -> A* baseline path
  -> postprocess corridor
  -> fallback_box / workspace_iris region sequence
  -> optional pydrake_direction_cone_program / pydrake_control_point_direction_cone_program
  -> sampled 2D geometric candidate
  -> collision / cost / direction_cone / motion feasibility diagnostics
  -> path-planner-route/v1
```

当前 GCS 方向的核心增量是 `direction_cone` 与 control-point derivative proxy。它在固定
`convex_region_sequence` 上使用 Drake `MathematicalProgram` 求解 2D waypoint
变量或 control point 变量，并对相邻 region seed edge 或 successive control-point
derivative proxy 添加方向锥线性约束。control-point backend 现在额外加入确定性的
terrain-cost proxy，用 region 内 inverse-cost weighted passable-cell centroid 作为
`control_point_terrain_anchor_quadratic` 目标项，并在 route JSON 中区分 solver proxy
与 sampled 后验 terrain cost。当前 backend 包括
`pydrake_direction_cone_program` 和 opt-in
`pydrake_control_point_direction_cone_program`，不是完整 GCS graph search，也不是
production Bezier/B-spline 轨迹优化器。

本项目当前不加入 `hard` 曲率 fallback。若 `direction_cone` 未评估、未实际
backend-enforced、存在 violation，或 sampled candidate 发生碰撞、运动风险不可接受、
成本劣化，候选路径必须被阻断，而不是静默替换 route。

## 3. 当前系统架构

| 层级 | 当前模块 | 职责 |
| --- | --- | --- |
| CLI / 契约层 | `path_planner.cli`, `adapters.py` | 读取 request，输出 `path-planner-route/v1`。 |
| 搜索层 | `search/` | 生成 A* baseline，提供稳定 reachable 与 path cost 语义。 |
| 后处理层 | `postprocess/` | 基于 baseline 生成 corridor 和 smoothed geometric path。 |
| 区域层 | `drake_backend/corridor_regions.py`, `iris.py` | 生成 `ConvexRegionSequenceReport`；当前默认可用边界是 `fallback_box`，可选 `workspace_iris` 只作为工作空间 H-polyhedron backend。 |
| GCS 原型层 | `drake_backend/gcs_trajectory.py`, `gcs_control_point_trajectory.py` | 在固定 region sequence 上求解 waypoint 或 control-point direction-cone prototype。 |
| 诊断层 | `drake_backend/gcs_diagnostics.py`, `gcs_motion_feasibility.py` | 输出 direction cone、terrain cost、high-cost exposure、curvature/heading/turning-radius 后验诊断。 |
| 候选选择层 | `drake_backend/gcs_candidate.py` | 只在约束、碰撞、运动诊断和成本门槛均通过时选择 GCS geometric candidate。 |
| CLI batch 证据层 | `drake_backend/gcs_cli_batch.py` | 通过真实 `path_planner.cli` route JSON 汇总 GCS direction-cone、motion-feasibility 与 control-point terrain-cost 场景证据。 |
| 测试层 | `tests/test_drake_backend.py` 等 | 覆盖 route additive fields、fallback reason、candidate blocking、可选 Drake backend。 |

## 4. 明确非目标

当前 `path-planner` 不使用 IrisNp、IrisNp2、IrisZo 或 C-space IRIS。文档中若提到
IRIS，只指本项目已有的 2D workspace/fallback region 诊断边界。

当前 `path-planner` 不实现完整 Graph of Convex Sets branch-and-bound、rounding、
graph search 或 fixed-path re-solve。`pydrake_direction_cone_program` 是固定 region
sequence 上的连续优化原型。

当前 `path-planner` 已有 opt-in control-point derivative proxy 原型，但不实现
production Bezier/B-spline trajectory optimizer。当前 control point 仍服务于 sampled
2D geometric candidate diagnostic。

当前 `path-planner` 不加入 `hard` 曲率 fallback，也不把 sampled geometric path 表述为
Ackermann-feasible trajectory。

## 5. `direction_cone` 约束设计

给定相邻 region seed `s_i, s_{i+1}`，waypoint backend 为每个 region 建一个 2D 连续变量
`p_i = [x_i, y_i]`；control-point backend 为每个 region 建一个 2D control point
`cp_i = [x_i, y_i]`。两者都添加：

- region containment：`A_i p_i <= b_i` 或 `A_i cp_i <= b_i`
- start/goal fixed：首尾 waypoint/control point 固定到首尾 seed
- direction cone per edge 或 derivative proxy：

```text
t^T d_i >= rho
n^T d_i - eta * t^T d_i <= 0
-n^T d_i - eta * t^T d_i <= 0
```

其中 waypoint backend 使用 `d_i = p_{i+1} - p_i`，control-point backend 使用
`d_i = cp_{i+1} - cp_i` 作为 derivative proxy。`t` 是相邻 seed 的参考切向，`n` 是法向，
`eta = tan(theta_max)`。`rho` 不再只作为一个不可解释常数；报告中需要说明它来自
seed distance、相邻 region portal/overlap width 和 support width 的保守下界组合。

`gcs_trajectory_constraint_summary` 应报告：

- `backend_enforced`
- `enforcing_backend`
- `solver_constraint_count`
- `eta`
- `rho_lower_bound_min_m`
- `rho_source_counts`
- `portal_width_min_m`
- `support_width_min_m`
- `constraint_tightness_min`
- `trajectory_parameterization`
- `control_point_count`
- `derivative_constraint_count`
- `control_point_region_containment_count`
- `objective_terms`
- `objective_term_weights`
- `terrain_objective_source`
- `terrain_objective_weight`
- `terrain_objective_boundary`
- per-edge/per-sample risk flags

只有 solver 实际添加 direction-cone 线性约束并成功求解时，才允许
`backend_enforced=true`。

## 6. 成本设计

当前 objective 保持简洁：

- segment length quadratic proxy
- low-cost anchor tracking proxy
- second-difference smoothness proxy
- control-point second-difference proxy
- control-point terrain anchor quadratic proxy

route JSON 的成本解释以 sampled path 后验计算为准：

- `path_length`
- `terrain_path_cost`
- `sampled_terrain_cost`
- `control_point_terrain_cost`
- `terrain_objective_source`
- `terrain_objective_weight`
- `terrain_objective_boundary=proxy_not_continuous_field_integral`
- `high_cost_exposure`
- `baseline_high_cost_exposure`
- `energy_proxy`
- `smoothness_proxy`
- `sampled_cell_count`
- `blocked_sample_count`
- baseline/postprocess cost delta
- candidate decision 和 quality gate

候选选择不能只看 solver success。若 candidate 未带来 baseline/postprocess 质量收益、
与 baseline 重复、碰撞、运动诊断 infeasible、high-cost exposure 相对 baseline/postprocess
变差，或 direction cone 未 enforce/violate，必须输出明确 fallback reason。

当前 terrain-cost objective 仍是工程代理：它使用固定 region sequence 中的 passable
cell cost 生成 control-point anchor，不是连续地形场积分，也不替代 sampled path 后验成本。

## 7. Ackermann 边界

月面巡视车后续当然需要车辆运动学、执行控制与闭环跟踪验证，但当前
`path-planner` 这一阶段只输出 2D geometric candidate 和 sampled diagnostics。

“不宣称 Ackermann-feasible trajectory”的含义是：

- 不声明当前输出已经满足前轮转角、轴距、速度/加速度连续性和控制输入约束。
- 不声明当前 sampled path 可直接由 Ackermann 底盘闭环执行。
- 当前 motion feasibility 只是曲率、heading、turning-radius 后验诊断，用来阻断高风险
  geometric candidate。

后续若要进入 Ackermann feasibility 层，需要新增独立模型、约束、控制接口和验证门槛，
不能由当前 `direction_cone` summary 自动推出。

## 8. 当前风险与后续阶段

当前主要剩余风险：

- control-point derivative proxy 仍弱于 production Bezier/B-spline trajectory optimizer。
- portal/support width 的 `rho` 参数已进入 route JSON 和 scenario summary，但仍是保守工程下界。
- terrain/high-cost 通过 control-point terrain anchor proxy 影响 solver，不是连续地形场积分成本。
- 可选 Drake backend 不可用时只能输出明确 not-evaluated summary，不能宣称 GCS 求解。

Portal/Cost Calibration、Scenario Matrix、`GCS Direction-Cone CLI Scenario Batch Evidence v1`
与 `GCS Motion-Feasibility Batch Gate v1` 已完成到 route JSON evidence 层。随后完成
`GCS Control-Point Direction-Cone Prototype v1` 与
`GCS Control-Point Terrain-Cost Calibration v1` 已推进到 `path-planner` route JSON
evidence 层：

- 新增 opt-in `pydrake_control_point_direction_cone_program`，用 successive control-point
  difference 作为 derivative proxy。
- 将 direction-cone 线性约束施加到 derivative proxy 上，并记录
  `trajectory_parameterization=control_point_derivative_proxy`、`control_point_count`、
  `derivative_constraint_count`、`control_point_region_containment_count` 和 `objective_terms`。
- 新增 `control_point_terrain_anchor_quadratic` 目标项、`objective_term_weights`、
  `terrain_objective_source`、`sampled_terrain_cost`、`control_point_terrain_cost` 和
  `terrain_objective_boundary=proxy_not_continuous_field_integral`。
- candidate gate 新增 high-cost exposure 不劣化检查：若总成本改善但 high-cost exposure
  相对 baseline/postprocess 变差，候选被阻断并输出明确 fallback reason。
- 通过 `--gcs-control-point-candidate` 从真实 `path_planner.cli` route JSON 验证 control-point
  backend、candidate decision、motion feasibility blocking、heading violation、
  turning-radius violation 和 pydrake unavailable。
- 新增 `--batch-kind control-point-terrain-cost`，输出
  `gcs_control_point_terrain_cost_cli_batch/v1`，覆盖 low-cost selected、cost-dominated、
  high-cost exposure blocked、motion-infeasible 和 pydrake-unavailable。
- 保留 `gcs_direction_cone_cli_scenario_batch/v1` 与 `gcs_motion_feasibility_cli_batch/v1`
  行为，避免旧证据合同回归。
- 保持 route contract additive，不改变 `trajectory_kind=geometric_path` 和 `reachable` 语义。

根仓库当前新增的是 `GCS Control-Point Semi-Real Evidence Bridge v1`：在
`scripts/run_path_feedback_validation.sh` 中显式 opt-in 转发
`--gcs-control-point-candidate`，让半真实 `cost/passable_mask` path-feedback validation
可以收集 control-point terrain-cost GCS 证据。这个 flag 不属于
`--diagnostic-profile all` 的默认语义，默认 acceptance gate 仍保持原有行为。

启用该 flag 后，`path-feedback-summary/v1` 和 batch summary 只新增 additive
`gcs_control_point_*` 字段，例如 report/attempt/success count、backend count、
candidate selected count、fallback reason count、terrain objective source count、
`sampled_terrain_cost` 统计、`high_cost_exposure_delta` 统计和 candidate audit。若
`pydrake` 不可用，summary 必须显示 `not_evaluated`，不能伪造为 GCS success。该桥接
不把 GCS candidate 当成默认 route 替代，不改变 `path-planner-route/v1`、
`model-explorer-contract/v1` 或 `path-feedback-summary/v1` 的既有稳定字段含义。

下一阶段新增的是 `GCS Control-Point Candidate Triage & Calibration v1`：summary
固定输出 `gcs_control_point_candidate_triage`，schema 为
`gcs-control-point-candidate-triage-summary/v1`；同时输出
`gcs_control_point_candidate_artifacts`，schema 为
`gcs-control-point-candidate-artifact-index/v1`。artifact index 为每个
control-point 候选记录 scenario/action、route JSON、request JSON、contract 和 sidecar
路径，使 `cost_dominated`、`direction_cone_constraint_violation`、
`unsupported_route_replacement` 等结果可以按候选复查。
`scripts/export_gcs_control_point_candidate_triage.py` 可把该 triage block 从
`path-feedback-summary.json` 导出为独立 JSON/Markdown 审查材料。

当前 evidence 根目录为 `outputs/path_feedback_gcs_control_point_triage_current/`；
独立导出文件为 `gcs-control-point-candidate-triage-summary.json` 与
`gcs-control-point-candidate-triage-summary.md`。该 run 的关键计数是：23 个
candidates、10 个 attempted、10 个 success、0 个 selected、23 个 route artifacts；
fallback counts 为 `cost_dominated=5`、`direction_cone_constraint_violation=5`、
`unsupported_route_replacement=13`；`calibration_sweep` 保持
`default_change_recommended=false`。
随后 `scripts/run_gcs_control_point_calibration_matrix.py` 将这些 request artifacts
按显式 solver 参数矩阵重跑，输出
`gcs-control-point-calibration-matrix-summary/v1`。当前矩阵 evidence 根目录为
`outputs/path_feedback_gcs_control_point_calibration_matrix_current/`：4 个 matrix
settings、23 个 candidates、92 个 route artifacts，`safety_regression_count=5`。
没有任何组合提高 selected count；`direction_cone_relaxed` 将
`direction_cone_constraint_violation` 从 5 降到 1，但 `cost_dominated` 从 5 升到
9，并出现 terrain cost regression，因此结论仍为
`recommendation=no_default_change_recommended`。

当前半真实 evidence 的解释边界是：

- smoke 组 control-point program 可以求解，但候选仍可能因 `cost_dominated` 或
  `direction_cone_constraint_violation` 被阻断。这表示质量门槛、terrain-cost proxy 和
  `direction_cone` 约束需要校准，不表示应放宽 safety gate。
- stress 组若 region sequence 不连通或 route replacement 不支持，`not_evaluated` 是预期
  诊断结果，不应被解读为 GCS 成功或失败后静默替换 route。
- triage 内含 `calibration_sweep`
  （`gcs-control-point-candidate-calibration-sweep/v1`），这是基于已记录候选的诊断
  sweep，覆盖 terrain objective weight、second-difference weight、direction-cone
  rho/eta/tolerance 和 quality-gate 阈值。它只给出下一轮 solver rerun 的实验矩阵；
  默认参数只有在 terrain cost、high-cost exposure、collision、`direction_cone` 和
  motion diagnostics 均不劣化时才可更新。
- 该阶段仍不实现完整 GCS graph search/rounding/branch-and-bound，不使用 C-space IRIS，
  不恢复 hard 曲率 fallback，不宣称 Ackermann-feasible trajectory。
