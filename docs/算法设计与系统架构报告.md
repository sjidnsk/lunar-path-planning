# `path-planner` 算法设计与系统架构报告

生成日期：2026-06-06

## 1. 资料边界

本报告面向根仓库当前使用的子项目 `path-planner`，而不是旧的
`a_gcs_ws-2.0.1-direction-cone-curvature` 实验目录。报告依据如下：

- `path-planner/src/path_planner/`
- `path-planner/tests/`
- `path-planner/docs/superpowers/specs/`
- 根仓库与 `path-planner` 之间的 JSON 契约

当前 `path-planner` 的执行层边界是 2D 几何路径规划与诊断。它可以评估 sampled
geometric candidate 的碰撞、地形代价、方向锥约束和曲率/航向风险，但不宣称输出
Ackermann-feasible trajectory。

## 2. 总体结论

`path-planner` 当前是一个分层路径规划与执行层评估原型。稳定主链路仍然是：

```text
path-planner-request/v1
  -> cost/passable grid
  -> A* baseline path
  -> postprocess corridor
  -> fallback_box / workspace_iris region sequence
  -> optional pydrake_direction_cone_program
  -> sampled 2D geometric candidate
  -> collision / cost / direction_cone / motion feasibility diagnostics
  -> path-planner-route/v1
```

当前 GCS 方向的核心增量是 `direction_cone`。它在固定
`convex_region_sequence` 上使用 Drake `MathematicalProgram` 求解 2D waypoint
变量，并对相邻 region seed edge 添加方向锥线性约束。该 backend 名为
`pydrake_direction_cone_program`，不是完整 GCS graph search，也不是
Bezier/B-spline 轨迹优化器。

本项目当前不加入 `hard` 曲率 fallback。若 `direction_cone` 未评估、未实际
backend-enforced、存在 violation，或 sampled candidate 发生碰撞、运动风险不可接受、
成本劣化，候选路径必须被阻断，而不是静默替换 route。

## 3. 当前系统架构

| 层级 | 当前模块 | 职责 |
| --- | --- | --- |
| CLI / 契约层 | `path_planner.cli`, `adapters.py` | 读取 request，输出 `path-planner-route/v1`。 |
| 搜索层 | `search/` | 生成 A* baseline，提供稳定 reachable 与 path cost 语义。 |
| 后处理层 | `postprocess/` | 基于 baseline 生成 corridor 和 smoothed geometric path。 |
| 区域层 | `drake_backend/corridor_regions.py`, `iris.py` | 生成 `ConvexRegionSequenceReport`；当前默认可用边界是 `fallback_box`，可选 `workspace_iris` 只作为工作空间 H-polyhedron backend。 |
| GCS 原型层 | `drake_backend/gcs_trajectory.py` | 在固定 region sequence 上求解 `pydrake_direction_cone_program`。 |
| 诊断层 | `drake_backend/gcs_diagnostics.py`, `gcs_motion_feasibility.py` | 输出 direction cone、terrain cost、high-cost exposure、curvature/heading/turning-radius 后验诊断。 |
| 候选选择层 | `drake_backend/gcs_candidate.py` | 只在约束、碰撞、运动诊断和成本门槛均通过时选择 GCS geometric candidate。 |
| 测试层 | `tests/test_drake_backend.py` 等 | 覆盖 route additive fields、fallback reason、candidate blocking、可选 Drake backend。 |

## 4. 明确非目标

当前 `path-planner` 不使用 IrisNp、IrisNp2、IrisZo 或 C-space IRIS。文档中若提到
IRIS，只指本项目已有的 2D workspace/fallback region 诊断边界。

当前 `path-planner` 不实现完整 Graph of Convex Sets branch-and-bound、rounding、
graph search 或 fixed-path re-solve。`pydrake_direction_cone_program` 是固定 region
sequence 上的连续优化原型。

当前 `path-planner` 不把 GCS 变量升级为 Bezier/B-spline control points。后续可以把
region waypoint `p_i` 替换为控制点，并把方向锥约束施加到导数控制点上，但这属于
future hook。

当前 `path-planner` 不加入 `hard` 曲率 fallback，也不把 sampled geometric path 表述为
Ackermann-feasible trajectory。

## 5. `direction_cone` 约束设计

给定相邻 region seed `s_i, s_{i+1}`，当前 backend 为每个 region 建一个 2D 连续变量
`p_i = [x_i, y_i]`，并添加：

- region containment：`A_i p_i <= b_i`
- start/goal fixed：首尾 waypoint 固定到首尾 seed
- direction cone per edge：

```text
t^T d_i >= rho
n^T d_i - eta * t^T d_i <= 0
-n^T d_i - eta * t^T d_i <= 0
```

其中 `d_i = p_{i+1} - p_i`，`t` 是相邻 seed 的参考切向，`n` 是法向，
`eta = tan(theta_max)`。`rho` 不再只作为一个不可解释常数；报告中需要说明它来自
seed distance、相邻 region portal/overlap width 和 support width 的保守下界组合。

`gcs_trajectory_constraint_summary` 应报告：

- `backend_enforced`
- `enforcing_backend`
- `solver_constraint_count`
- `eta`
- `rho_lower_bound_min_m`
- `rho_source_counts`
- `portal_width_min_m`
- `support_width_min_m`
- `constraint_tightness_min`
- per-edge/per-sample risk flags

只有 solver 实际添加 direction-cone 线性约束并成功求解时，才允许
`backend_enforced=true`。

## 6. 成本设计

当前 objective 保持简洁：

- segment length quadratic proxy
- low-cost anchor tracking proxy
- second-difference smoothness proxy

route JSON 的成本解释以 sampled path 后验计算为准：

- `path_length`
- `terrain_path_cost`
- `high_cost_exposure`
- `energy_proxy`
- `smoothness_proxy`
- `sampled_cell_count`
- `blocked_sample_count`
- baseline/postprocess cost delta
- candidate decision 和 quality gate

候选选择不能只看 solver success。若 candidate 未带来 baseline/postprocess 质量收益、
与 baseline 重复、碰撞、运动诊断 infeasible，或 direction cone 未 enforce/violate，
必须输出明确 fallback reason。

## 7. Ackermann 边界

月面巡视车后续当然需要车辆运动学、执行控制与闭环跟踪验证，但当前
`path-planner` 这一阶段只输出 2D geometric candidate 和 sampled diagnostics。

“不宣称 Ackermann-feasible trajectory”的含义是：

- 不声明当前输出已经满足前轮转角、轴距、速度/加速度连续性和控制输入约束。
- 不声明当前 sampled path 可直接由 Ackermann 底盘闭环执行。
- 当前 motion feasibility 只是曲率、heading、turning-radius 后验诊断，用来阻断高风险
  geometric candidate。

后续若要进入 Ackermann feasibility 层，需要新增独立模型、约束、控制接口和验证门槛，
不能由当前 `direction_cone` summary 自动推出。

## 8. 当前风险与后续阶段

当前主要剩余风险：

- waypoint 变量比 Bezier/B-spline derivative control point 更弱。
- portal/support width 的 `rho` 参数仍是保守工程下界，需要更多场景校准。
- terrain/high-cost 通过 anchor proxy 影响 solver，不是连续地形场积分成本。
- 可选 Drake backend 不可用时只能输出明确 not-evaluated summary，不能宣称 GCS 求解。

下一阶段应继续围绕 `GCS Direction-Cone Portal and Cost Calibration v1` 推进：

- 用更多窄通道、高代价绕行、退化 portal、collision、motion infeasible fixture 验证阻断逻辑。
- 校准 `rho`、`eta`、portal/support width 和 cost summary 的解释能力。
- 保持 route contract additive，不改变 `trajectory_kind=geometric_path` 和 `reachable` 语义。
